# LiveMTG UI再設計 仕様書

作成: 2026-07-16 ／ 対象: `index.html`（約2,000行・単一ファイルSPA）
ステータス: **全7ステップ実装完了**（2026-07-16）。追加実装済み: 事前メモ取り込み・解釈の確認（ライブ一問一答）・カスタムセレクト。UXレビュー指摘の3点も実装済み（同日）: ①狭幅レールのオーバーレイ化 ②レール内の補足・訂正チャット（§5.3-4） ③録音中の停滞バッジ（§3.2）。残タスク: 実会議での通しテスト → `dist/npm` パッケージ版への反映

## 0. 目的

LiveMTGの中心体験を「**録音しながら、会話を把握し、次の一手を得る**」に戻す。
現状の問題は機能不足ではなく配置の問題：

- AI提案（次の質問・ガイド・AIサポート）が縦積みで、メイン表示（議事カード）を画面外へ押し出す（`index.html` の `#nextwrap` → `#gwrap` → `#assist` → `.grid` の順の縦積み）
- ヘッダーの `#st`・`#lastupdate` が数秒ごとに書き換わりチカチカする
- 準備系（新規会議モーダル・目標ガイド設定モーダル・コパイロット）が3箇所に分散
- 成果物系（清書・マインドマップ・スライド・PDF・学び）がメニューの奥に埋まっている
- 「リスト／マインドマップ」の二段構造の中に4サブモードが隠れている

**方針: 機能は捨てない。置き場所だけ変える。**

## 1. スコープ

### やること
- 3フェーズ構造（準備／ライブ／振り返り）の導入
- ヘッダー簡素化と状態ピル化
- ライブ支援レール（右サイドパネル）の新設
- 表示モード5タブの同列化
- 成果物の振り返り画面への集約

### やらないこと（今回の非対象）
- **ダッシュボード画面の新設はしない**。会議切替は既存の `#sess` ドロップダウンを維持する。フェーズ構造が安定した後の別仕様とする
- フレームワーク導入・ファイル分割はしない。単一 `index.html` を維持する
- `server.py` のAPI変更はしない（UIの組み替えのみ）
- 録音まわりのロジック（`startCapture` / `doStart` / `doStop` / `sendChunk` / MediaRecorder関連）には**触れない**。直近で低遅延録音を検証済みのため、DOM移動もイベント再配線もしない

## 2. 全体構造 — 3フェーズ

### 2.1 フェーズ定義

| フェーズ | 役割 | 主な内容 |
|---|---|---|
| **準備** | 会議前 | 会議名・用途・立場・目標・背景フォルダ、コパイロット準備チャット、URL調査、準備会話の録音、マイク・AI動作確認 |
| **ライブ** | 会議中 | 5種の表示モード＋ライブ支援レール（次の質問・AIサポート・補足訂正） |
| **振り返り** | 会議後 | 話者確認、清書、マインドマップ成果物、スライド、1ページPDF、学び抽出、会議データ削除 |

### 2.2 実装方式（重要）

- 別ページ化・DOM再構築はしない。**`<body>` に `phase-prep` / `phase-live` / `phase-review` クラスを付け、CSSで表示切替**する
- 既存要素のIDは全て温存する。イベントリスナーの再配線を発生させない
- フェーズ状態は `localStorage`（キー: `livemtg_phase_<会議ID>`）に保持し、リロードで復元する

### 2.3 フェーズ遷移ルール（ウィザードにしない）

- フェーズタブはヘッダー直下に常設し、**いつでも自由に行き来できる**。一方通行の遷移は禁止
- 自動遷移は「提案」まで。強制しない：
  - 録音開始 → ライブへ自動切替（これは強制でよい。録音中に準備画面のままは事故のもと）
  - 録音停止 → 「振り返りへ移動しますか？」のトースト提案のみ。雑談から本題が再開するケースがあるため自動では移らない
- **録音ボタンはどのフェーズでも押せる**（ヘッダー常設。§3参照）
- 振り返り中の話者名修正・誤認識訂正は、ライブ中でも支援レールの補足・訂正チャットから実行できる（機能の重複配置を許容する）

### 2.4 新規会議のデフォルトフェーズ

- 新規会議作成直後 → 準備
- 既存会議を開いた時 → 最後にいたフェーズ（`localStorage`）。録音中なら常にライブ

## 3. ヘッダー仕様

### 3.1 常設要素（これ以外は置かない）

左から:

1. LiveMTGロゴ（既存 `.logo`）
2. 会議切替ドロップダウン `#sess`（ダッシュボード代替。将来ダッシュボードができたら「戻る」ボタンに置換）
3. フェーズタブ（準備｜ライブ｜振り返り）※ヘッダー内に収まらない場合はヘッダー直下の第2段に置く
4. 録音ボタン `#rec`（現位置・現ロジックのまま）
5. **状態ピル**（新設。§3.2）
6. その他メニュー `#menubtn`（残す項目は§3.3）

### 3.2 状態ピル（`#st` / `#lastupdate` / `#dot` の置換）

固定幅（`min-width` 指定・内容変化で幅が動かない）のピル1個に集約する。状態は4つのみ:

| 状態 | 表示 | 判定（既存ロジック流用: index.html:1045 付近） |
|---|---|---|
| 準備OK | ● 準備OK（緑） | サーバ接続済み・非録音・キュー空 |
| 録音中 | ● 録音中（赤・点滅は dot のみ） | `capturing \|\| serverRecording` |
| 処理中 | ● 処理中（青） | 非録音かつ `queue>0 \|\| analyzing \|\| detailing \|\| researching>0 \|\| exploring` |
| 要確認 | ● 要確認（黄） | サーバ断（既存 `#down` 表示条件）・マイクエラー・更新停滞（`lastupdate.stale` 相当） |

優先順位（実装で確定）: **サーバ断 > 録音中 > 処理中 > 更新停滞（非録音時のみ要確認）> 準備OK**。録音中に処理が走っていても「録音中」を出す（録音継続が最重要情報のため）。更新停滞は録音中には正常なラグなので、録音中は「要確認」に落とさない

- **録音中の停滞バッジ（2026-07-16 UXレビューで追加）**: 録音中はピルが「録音中」を出し続けるため、解析・文字起こしが完全に死んでも気づけなかった。正常なラグと区別する**3分閾値**で、(a) 文字起こし停滞（キュー滞留＋transcript 3分更新なし）または (b) 解析停滞（新しい発言が3分以上未反映）を検知したら、ピル右上に黄色の「!」バッジ＋ポップオーバーに注意行＋初回のみトースト。表示自体は「録音中」のまま（録音継続情報を失わない）

- **テキストの毎秒更新は禁止**。状態が変わった時だけ書き換える
- ピルをクリックすると小さな詳細ポップオーバーを開き、そこで初めて詳細を出す: 最終解析時刻（現 `#lastupdate` の内容）、AIプロバイダ名（Claude/Codex）、キュー件数、録音健全性（現 `#recordhealth` の内容）
- `#down`（サーバ停止バー）は安全情報なので現状どおり全幅バーで残す

### 3.3 その他メニューに残す項目

- 新規会議 `#new`
- プロフィール `#profbtn`
- AI・音声の接続診断 `#runtimebtn`
- 言語 `#language`
- この会議を削除 `#delete`（会議管理の操作なのでメニューに残す。振り返り画面には置かない ※§6.4の決定事項）

メニューから**消す**項目（各フェーズ画面へ移動）: `#full` `#goalbtn` `#final` `#slides` `#regen` `#deckbtn` `#deckregen` `#pdf` `#learnbtn`

## 4. 準備フェーズ仕様

現在3箇所に分散している準備系UIを1画面に統合する。

### 4.1 構成（1カラム、上から）

1. **会議設定カード** — 会議名・用途 `#mtype` 相当・立場 `#mstance` 相当・目標 `#mgoal` 相当・背景フォルダ `#mproj` 相当。現在の「新規会議モーダル `#mbg`」と「商談ガイド設定モーダル `#gbg`」は**同じ項目の二重実装**なので、この画面のインライン編集フォームに一本化する（保存APIは既存のまま）
2. **準備チャット** — 現コパイロットモーダル `#sbg` の中身（クイックボタン `.prep-quick`・チャットログ `#slog`・入力 `#sinput`・準備ボード `#sboard`）をモーダルからこの画面へ**移設**。URL調査・相手情報共有はここ
3. **準備会話の録音** — 現 `#preprec` ボタンをここに置く
4. **動作確認カード** — マイク選択（現 `#micsel` 相当の一覧＋レベルメーター）と「AI応答テスト」（現 `#aitest`）への導線。接続診断モーダル `#diagbg` 自体は残し、ここからも開けるようにする

### 4.2 モーダルの扱い

- `#mbg`（新規会議）: **残す**。ただし項目は会議名＋用途のみに減らし、「詳細は準備画面で」に誘導。作成後すぐ準備フェーズが開くため入口は薄くてよい
- `#gbg`（商談ガイド設定）: **廃止**。準備画面の会議設定カードに吸収
- `#sbg`（コパイロット）: モーダルとしては廃止。中身は準備画面（§4.1-2）とライブ支援レール（§5.3）に分割配置。フローティングバブル `#copilotbubble` は廃止

## 5. ライブフェーズ仕様

### 5.1 レイアウト

```
┌ ヘッダー（§3） ──────────────────────────────┐
├ 準備｜▶ライブ｜振り返り ──────────────────────┤
├ リスト｜論点マップ｜放射マップ｜会話の関係｜時系列 ─┤
│                                    ┃ ライブ支援   │
│   選択中のメイン表示                ┃ レール       │
│   （タブに応じて .grid / #deck）    ┃ （§5.3）    │
│                                    ┃              │
└────────────────────────────────────┴──────────────┘
```

- CSS Grid の2カラム（`main` + `aside`）。レール幅 320〜380px
- `#summary`（今の要約1行）と `#who` / `#livespeakers`（話者バー）はメイン表示の上部に残す

### 5.2 表示モード5タブ（同列化）

現在の二段構造（`#viewseg` の list/slide ＋ `mindmapMode` の tree/radial/relation/timeline）を廃止し、5つを同列タブにする:

| タブ表示名 | 既存の実体 | 備考 |
|---|---|---|
| リスト | `view='list'`（`.grid` のカード群） | |
| 論点マップ | `mindmapMode='tree'` | 現UI名「マインドマップ」から改名 |
| 放射マップ | `mindmapMode='radial'` | |
| 会話の関係 | `mindmapMode='relation'` | **UIから「Mermaid」表記を排除**（index.html と make-mindmap.py:140 のボタンラベル両方） |
| 時系列 | `mindmapMode='timeline'` | |

- 内部状態は既存の `view` / `mindmapMode` / `activeView()`（index.html:1290）をそのまま使う。タブUIだけ一段にする
- `localStorage` キー（`livemtg_mindmap_mode` 等）と `sendViewFocus()` の挙動は変更しない
- **用語の区別**: ライブ表示のタブは「論点マップ」、振り返りで生成する成果物（`mindmap.html`）は「マインドマップ（成果物）」と呼び分ける

### 5.3 ライブ支援レール（右サイド・新設）

**2026-07-16 変更（依頼者指示）: レール内は縦積みではなく2タブ構成にする。**

- タブ1「質問・参考回答」= 次に聞く質問＋疑問への参考回答（会話中に即使う打ち手系）
- タブ2「会議の狙い」= 目標への進捗・回答の読み・調査結果（俯瞰する戦略系）
- 選択タブは `localStorage: livemtg_rail_tab` に保持。裏タブの内容が更新されたらタブに赤ドット（表示中タブには出さない・タブを開くと消える）
- 各タブに空状態のヒント文を置く（質問側：録音が進むと表示される旨／狙い側：目標設定への導線）
- **レール内のセクションは常時展開・折りたたみなし**（2026-07-16 依頼者指示。タブ分けで各面の情報量が減ったため「パッと見える」ことを優先。「会議の狙い」の畳んだバー＋クリック展開も廃止し、狙い本文は省略なしで全文表示。当初仕様の「各セクション個別折りたたみ」は撤回）
- **質問の意図・疑問への回答も常時表示**（2026-07-16 依頼者指示。旧UIの「チップをタップで意図／回答を展開」を全廃）。「Web検索で確認」は各カード内のテキストボタン（結果はカード内 `#vres-<i>`）
- **レールのデザイン言語（2026-07-16 確定。初版カードが「ダサい・見づらい」との指摘を受け再設計）**: 「小さなグレー見出し（12px・letter-spacing）＋フラットな白カード（1pxヘアライン `var(--line)`・radius 14px・薄い影）＋文字階層（主文14px ink／補足13px sub）」のみで構成する。**左アクセント帯・色付き背景・カード内ラベル・アイコン・外箱の入れ子・黒バーは使わない**。**絵文字も全体で使用禁止**（2026-07-16 依頼者指示。✓ ✗ ▾ › などのモノクロ記号は可）。「会議の狙い」も黒グラデバーを廃止して同じ言語（見出し＋狙い本文の白カード＋gpanel白カード群）に統一。**レール自体は画面右端まで伸びるフラットなサイドバー面（`#eceef2`＋左ヘアライン1本のみ。角丸・囲い枠なし）として描き、メイン表示（ページ地 `#f5f5f7` に直置き）と面の差で区別する**（2026-07-16 依頼者指示。角丸パネルで囲う案は「カードみたいに囲うのは好きじゃない」で却下→端まで伸びるサイドバーに変更。実装は `aside.siderail`＝色面、内側 `.railinner`＝sticky＋スクロール）

レールの構成要素:

0. **解釈の確認（2026-07-16 依頼者発案で追加）** — 清書前確認の「一問一答でAIの解釈を潰す」体験のライブ版。**⚠実装経緯の教訓（2026-07-16夜判明）**: 当初 `INCR_PROMPT` に定義したが、解析の並列レーン分割で `INCR_PROMPT` 自体が未使用の死にコードになっており、confirmは**一度も出力されていなかった**。現在は即時レーン `LIVE_PATCH_PROMPT` に移植済み（発火条件を「①曖昧な固有名詞の同定 ②担当の対応 ③金額・数量・期日 ④初出の略語」の4種に限定し、「言われなかった詳細を確認事項にしない」抑制ルール付き。haiku実機で陽性・陰性の両ケースを検証済み）。**新しい出力フィールドはレーン側プロンプト（LIVE_PATCH/DETAIL_PATCH/ACTIVE_*）に追加すること**。さらに同日、「LLMは自信のなさを自己申告できない（過信バイアス）」対策として**機械検出を追加**（依頼者指示「物珍しい単語や人名はとりあえず確認にあげる」）：`_mech_confirms()` が文字起こし差分から ①既知情報（学習用語・プロフィール・参加者・ライブ補足）に無いカタカナ語（4字以上・一般語ストップリスト除外・transcript内2回以上出現） ②「◯◯さん/様/氏」の未知人名 を抽出し、AIを介さず確認カード化する（1チャンク最大2件・会議ごとに `confirm-raised.json` で再提起防止・**AI解析が失敗していても届く**）。ライブ解析が自信の持てない解釈を `confirm[]`（最大4件・断定形一文＋根拠）として出し、レールの「質問・参考回答」タブに「✓合ってる／✗違う」カードで表示。合ってる→確認済みとしてライブ補足へ送信、違う→インライン入力で訂正を送信（どちらも `/api/live-notes` 経由で次の解析に最優先反映、プロンプト側で「確認・訂正済みの点は二度と出さない」）。回答済みは会議ごとに `localStorage: livemtg_confirm_done_<id>` で非表示。ポーリング再描画で訂正入力中のテキストが消えないよう署名ガード付き
1. **次に聞く質問** — 現 `#nextwrap` / `#nextchips` を移設。タップで意図表示は現行どおり ✅改名済み（旧「今、次にするべき質問」はレール幅で折り返すため短縮）
2. **会議の狙い** — 現ガイド `#gwrap` / `#gbody` を移設。折りたたみ挙動（`#ghead` トグル）は現行どおり ✅改名済み（旧「ガイド」）
3. **疑問への参考回答** — 現 `#assist` / `#achips` / `#apanel` を移設。「調べて」もここ ✅改名済み（旧「AIサポート」。「会議の狙い」との役割の違いが名前で分からないという指摘を受け、2026-07-16 決定：狙い＝目標への進捗と回答の読み／参考回答＝会話中に出た疑問へのAI調査回答）
4. **補足・訂正チャット** — コパイロットのチャット部（`#slog` / `#sinput` 系）のライブ版。「AIの勘違いを訂正」「背景情報を追加」クイックボタン付き ✅実装済み（2026-07-16）: `#railfix` としてレール**最下部に常駐**（どちらのタブからも使えるようタブ非依存・`margin-top:auto` で下端固定）。1行入力＋クイックボタン（「訂正：」「補足：」を先頭挿入）＋直近2件の送信済み表示。送信は `/api/live-notes`（解釈の確認と同じ経路）で次の解析に最優先反映。Enter送信はIME変換確定と衝突しないよう `isComposing` ガード付き。会議切替で送信済み表示を読み直す

要件:

- **どの表示タブを選んでいてもレールは常時同じ**。タブ切替でレールが消えたり内容が変わったりしない
- 各セクションは個別に折りたたみ可。折りたたみ状態は `localStorage` に保持
- **レール全体の折りたたみボタン**を必ず付ける（Zoom/Meetと画面を分け合う状況が通常運用のため）。折りたたみ時はメイン表示が全幅になる
- レスポンシブ: ビューポート幅 **1100px未満** でレールを**右からのオーバーレイ**に切替える（2026-07-16 UXレビューで「メイン下部落とし」から変更。Zoom等と半画面を分け合う通常運用でレールが埋もれ、畳む手段も失われていたため）。開閉トグル `#railtoggle` は狭幅でも常時表示。レールを畳んでいる間に新着が届いたらトグルに赤ポチ（開くと消える）。オーバーレイの上端はヘッダー実高に追随（`--hdrh` をJSで同期。900px未満でヘッダーが折り返して58pxを超えるため）
- 新しい質問・サポートが届いた時は**レール内のバッジ**で通知。メイン表示を押し下げるレイアウトシフトは一切禁止

### 5.4 会話全文

- 現行のドロワー（`#drawer` / `#drbody`）を維持。開くボタンはメイン表示タブ列の右端に「全文」として置く（メニューから移動）

## 6. 振り返りフェーズ仕様

### 6.1 構成

### 6.2 生成中の挙動

- 各カードに進捗表示（既存のポーリング状態を流用）と**キャンセルボタン**（既存 `cancelCurrentOperation()` を接続）を置く
- 生成中でも他フェーズへ移動できる。移動しても処理は継続し、状態ピルが「処理中」になる

### 6.3 推奨フロー表示

カード順＝推奨順（話者確認 → 清書 → 成果物 → 学び）とし、番号を振る。ただし順序は強制しない（清書せずPDFだけ欲しいケースあり）

### 6.4 決定事項: 会議データ削除の置き場

**その他メニューのみ**に置く（§3.3）。振り返り画面には置かない。理由: 振り返りは「成果物を作る」画面であり、破壊的操作を成果物ボタンの隣に並べると誤タップリスクがある
