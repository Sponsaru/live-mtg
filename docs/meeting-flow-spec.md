# LiveMTG 会議進行コパイロット仕様書

- 作成日: 2026-07-21
- ステータス: Draft 2（2026-07-21 レビュー反映）
- 対象: 準備フェーズ、ライブフェーズ、会話可視化、会議後の整理
- 関連文書: `ui-redesign-spec.md`

## 1. 目的

LiveMTGを「会議内容をリアルタイムで見返せる議事録」から、「会議中に聞き漏らし・脱線・未決定を減らし、着地点まで進める会議進行コパイロット」へ拡張する。

ユーザーはメイン画面だけで次を判断できること。

1. 今日どこまで進めたいか
2. 現在どの議題を話しているか
3. 次に何を聞くべきか
4. 何が回答され、何が決まったか
5. 何が未解決で、何を持ち越すか

## 2. 中心体験

```text
準備の壁打ち
  ↓
着地点・議題・質問の会議設計へ構造化
  ↓
本番会議へそのまま引き継ぐ
  ↓
会話から現在の議題・回答・結果・未確定事項を更新
  ↓
次の質問・追加議題・打開案をAIが提案
  ↓
議題の進捗と優先順位を更新
```

## 3. 情報設計

### 3.1 会議の着地点

「会議の狙い」と「今日の着地点」はユーザーに二重入力させず、メイン画面では「今日の着地点」に統合する。

保持する情報:

- 着地点の一文
- 成功条件（任意）
- 全議題の進捗要約
- 未完了議題数
- 残り時間（取得できる場合）

### 3.2 議題

議題は「話題」ではなく、会議中に回答・判断・方向付けのいずれかを進める単位とする。

状態:

- `not_started`: 未着手
- `discussing`: 議論中
- `discussed`: 話し終わり
- `deferred`: 議題自体を次回へ持ち越し

これとは別に合意状態を持つ。

- `not_applicable`: 合意対象外（情報共有・ヒアリング等）
- `pending`: 未合意
- `agreed`: 合意済み
- `rejected`: 見送り

進行状態と合意を混在させない。同じ議題が `discussed + pending`
になることも、`discussing + agreed` になることもある。録音中に
`discussing` になる議題は常に1件以下とし、話題が移れば直前の議題を
`discussed` にする。録音停止・会議切替時に現在議題を閉じるが、
合意状態は変えない。回答の進み具合は質問の回答数と結果の
「回答されたこと」で、宿題化は結果の「次の行動」で表す。

状態はAIが推定更新できるが、ユーザーは常に手動修正できる。上書きの可否は「6.3 AIの操作境界」に従う。

### 3.3 質問

質問は議題と別エンティティとして保持し、原則としていずれかの議題に紐づける。

状態:

- `next`: 次に聞く質問
- `queued`: 後で聞く質問
- `asked`: 質問したが回答判定待ち
- `answered`: 回答あり
- `deferred`: 次回・宿題へ持ち越し
- `dismissed`: 不要・却下

採用前のAI提案かどうかは議題と同じく `approval` フィールドで表す。`status` は進行のみを表し、採用状態を二重に持たない。

既存議題を前進させる確認は質問とする。複数の論点、意思決定、役割分担や期限設定が必要な質問は、AIが議題への昇格を提案する。

紐づく議題が判定できない質問は「未分類の質問」として保存し、破棄しない。

### 3.4 結果

議題の結果は次の4種類で表示する。

- 回答されたこと
- 決まったこと
- 次の行動（担当・期限を含む）
- 未解決・要確認

#### 「結論を確定」の扱い

メイン画面に汎用的な「結論を確定」ボタンは置かない。

理由:

- 何をもって「確定」とするかが不明瞭
- 議題の完了と、記載内容の真偽確認が混ざる
- 議題ごとにクリックを必須にすると会議中の操作負荷が高い

代わりに次の動作とする。

1. AIが発言根拠とともに「現在の結果」を自動更新する
2. AI整理の結果には「AI整理」と更新時刻を表示する
3. ユーザーは「結果を修正」から内容を直せる
4. ユーザーが修正した結果の項目はロックし、AIはその項目を上書きしない。新しい回答・決定・行動の追加は継続する
5. 議題の詳細から「議論の状態」と「合意の状態」を別々に変更できる

AIは明示的な合意発言がある場合に限り「決まったこと」へ追加できる。根拠が弱い場合は「現在の方向」または「未解決」とし、決定を創作しない。

### 3.5 AI提案と支援情報

支援情報は次の種類に分ける。

- `agenda_proposal`: 議題の追加・分割・統合・順番変更案
- `question_proposal`: 追加質問・深掘り質問
- `research`: AIの調査結果と出典
- `unstuck`: 行き詰まり時の別の聞き方・選択肢・仮置き案
- `warning`: 矛盾、聞き漏らし、残り時間に対する未着手議題

AI提案は正式な議題や質問へ自動混入させない。「議題に追加」「質問に追加」「今使う」「保留」「不要」の操作を持たせる。

AIが会話から新規抽出した仮議題・仮質問は、それぞれ `agendas` / `questions` に `approval=draft` で保存し、メインで直接扱えるようにする。採用済み議題の削除・分割・統合・順番変更など、既存構造を変える案は `suggestions` に保存する。

## 4. 画面仕様

### 4.1 ライブメイン画面

```text
┌ 固定ヘッダー: 会議名 / 録音 / 状態 / その他 ────────────────┐
│                                                                    │
│ 今日の着地点                                                     │
│ 料金体系と導入時期を決める                                     │
│ 議題 1/3完了・残り28分                                           │
│ [整理] [論点マップ] [放射マップ] [会話の関係] [時系列]               │
│                                                                    │
│ ▼ 料金体系を決める         ● 議論中     質問 2/4                  │
│    次に聞く: 最低契約期間はありますか？                           │
│    回答済み: 初期費用は発生しない                               │
│    現在の結果: 月額制が有力。利用人数は未確認                         │
│    AI提案 1件                           [詳しく見る]             │
│                                                                    │
│ ▶ 導入時期を決める         ○ 未着手     質問 0/2                  │
│ ▶ 運用体制を決める         ○ 未着手     質問 0/3                  │
│                                                                    │
│ [+ 議題を追加]                                                   │
└─────────────────────────────────────────────────────────────────────────┘
```

ライブフェーズの固定フッターは廃止し、会話可視化の入口は「今日の着地点」の中へ集約する。安全上常時表示が必要な録音状態とサーバー状態は固定ヘッダーで維持する。

### 4.2 今日の着地点セクション

- ライブ画面の最上部に表示する
- 着地点はインライン編集できる
- AIが推定した着地点は「AI仮作成」と表示する
- 会話可視化5ボタンを常時表示する
- 狭幅時はボタンを2段に折り返す

### 4.3 議題セクション

議題はセクショントグル（アコーディオン）とする。

閉じた状態で表示する情報:

- 並べ替えハンドル
- 議題名
- 状態
- 回答済み質問数 / 全質問数
- 結果の1行要約
- AI提案の未読数

開いた状態で追加表示する情報:

- 次に聞く質問
- 後で聞く質問
- 回答済み質問と回答要約
- 決まったこと
- 次の行動
- 未解決事項
- AI提案の概要
- 「詳しく見る」操作

開閉ルール:

- 複数議題の同時展開を許可する
- 初期状態は「現在の議題」のみ開く
- 完了・見送り議題は初期状態で閉じる
- AIが現在の議題を変えた場合、ユーザーが手動操作していない時だけ新しい議題を自動展開する
- ユーザーが別議題を開いたら自動追従を一時停止し「現在の議題に戻る」を表示する

並べ替え:

- デスクトップはドラッグハンドルを使う
- モバイル、キーボード、スクリーンリーダー用に「上へ」「下へ」「最初へ」「最後へ」を用意する
- AIによる順序変更は直接適用せず提案として表示し、承認時にまとめて適用する（「6.3 AIの操作境界」参照）

### 4.4 会話可視化ボトムシート

「今日の着地点」の各ボタンを押すと、画面下からボトムシートを開く。画面遷移はしない。

対象ビュー:

| UI名 | 現行内部表現 |
|---|---|
| 整理 | `view=list` |
| 論点マップ | `mindmapMode=tree` |
| 放射マップ | `mindmapMode=radial` |
| 会話の関係 | `mindmapMode=relation` |
| 時系列 | `mindmapMode=timeline` |

動作:

- 初期高さはビュポートの76%
- 上端ハンドルで高さを変更できる
- 全画面表示と閉じる操作を持つ
- `Escape`、下方スワイプ、背景クリックで閉じられる
- シート内で5ビューを切り替えられる
- 閉じた時はメイン画面のスクロール位置と議題開閉状態を維持する
- シートの開閉で録音・文字起こし・AI更新を停止しない

表示時は前回キャッシュを即座に出し、新しい会話がある場合だけ差分更新する。ゼロからの生成待ちで空画面にしない。

### 4.5 右ドロワー

右ドロワーは常設ナビゲーションではなく、選択対象を詳細表示するインスペクターとする。通常は閉じる。

開く操作:

- 議題の「詳しく見る」
- 質問の選択
- 結果の根拠発言の選択
- AI提案・調査結果・打開案の選択
- 時系列の発言の詳細表示

表示内容:

- 議題の全質問と回答
- 結果の根拠発言
- AIの判定理由
- 関連する調査結果と出典
- 議題・質問・結果の編集
- AI提案の採用・保留・却下

表示幅は `clamp(420px, 36vw, 560px)` とする。狭幅時は全画面に近いオーバーレイにし、ボトムシートと同時表示しない。

## 5. 準備フェーズ

### 5.1 壁打ちと会議設計の同時更新

準備チャットの右側に「会議設計」を表示する。AIの応答ごとに次を差分更新する。

- 今日の着地点
- 議題
- 議題に紐づく質問
- 未分類の質問
- 仮説
- 懸念・制約
- 読み込んだ資料と根拠

反映ルール:

- 「議題にしたい」「聞きたい」など、ユーザーが明示した内容は正式追加する
- AIが文脈から推測した議題・質問は「AI仮作成」とする
- ユーザーの明示内容をAIの再提案として重複生成しない
- すでに却下された提案を同じ根拠で再提案しない
- 新しく追加・変更された箇所は一時的に強調表示する

### 5.2 準備から本番への引き継ぎ

会議開始時にコピーや変換を行わない。準備と本番は同じ会議進行データを参照する。

ライブ画面を開いた時に次を表示する。

> 準備から議題3件・質問8件を引き継ぎました

未採用のAI仮議題・仮質問も破棄せず、仮提案の表示を保ったまま引き継ぐ。

## 6. 会議中のAI更新

### 6.1 共通データの常時更新

表示ビューと無関係に、次を軽量な進行解析レーンで更新する。

- 現在の議題
- 議題の状態
- 質問したか
- 回答が得られたか
- 回答要約
- 決まったこと
- 次の行動
- 未解決事項
- 追加質問・追加議題の提案

更新は単純な一定間隔ではなく、新しい意味のある発言が増えた時にデバウンスして実行する。短い相槌、無音、文字起こしの重複だけで更新しない。

### 6.2 議題未設定時

議題がない場合、会話の意味が判別できる量に達した後、AIが最大3〜5件の仮議題を作る。

- 挨拶と短い雑談だけで作らない
- 仮議題にも進捗追跡を開始する
- 「AI仮作成」と明示する
- ユーザーは採用、編集、統合、却下できる
- 数分後の会話から同一議題を別名で重複追加しない

何文字・何秒で生成を開始するかは調整可能な設定値とし、実会議テストで決定する。

### 6.3 AIの操作境界

AIが自動で行えること:

- 仮議題・仮質問の作成
- 現在の議題の推定
- 議題状態の推定
- 回答・結果・未確定事項の要約
- 発言根拠の紐づけ
- 不足質問、追加議題、順番変更、打開案の提案

AIが自動で行ってはいけないこと:

- ユーザーが作った議題の削除
- ユーザーが決めた議題順の変更
- ユーザーが編集・ロックした着地点、結果、状態の上書き
- 根拠発言のない回答済み判定
- 根拠発言のない決定事項の作成
- AI提案の正式議題・正式質問への無断追加

### 6.4 行き詰まり判定

下記の複数が成立した時に限り、`unstuck`提案を作る。沈黙だけで発火しない。

- 同じ論点が繰り返されている
- 質問したが回答が得られていない
- 新しい結果・回答・決定が増えていない
- 意見が複数案に分かれたまま収束しない
- 残り時間に対して未着手議題が多い

提案はメインの議題内に1行通知し、詳細は右ドロワーで表示する。自動で大きく割り込まない。

## 7. 会話可視化の更新設計

共通データの常時更新は「6.1」で規定する。可視化は次の優先順位で更新する。

1. 録音保全と文字起こし
2. 議題・質問・回答・結果の更新
3. 現在開いている可視化
4. 閉じている可視化

ビュごとの更新:

| ビュー | 更新方法 |
|---|---|
| 時系列 | 文字起こし到着時にAIを待たず追加 |
| 整理 | 共通データから即時描画し、進行解析で差分更新 |
| 論点マップ・放射マップ・会話の関係 | 前回キャッシュを即時表示し、表示中のみ差分更新 |

ボトムシートには常に次を表示する。

- 最終更新時刻
- 新しい会話の未反映有無
- 更新中・更新失敗・最新の状態
- 表示中ビュー名

ポーリング再描画でスクロール位置、マップ展開状態、選択ノード、議題の開閉を失わない。

## 8. 永続化データ

### 8.1 保存ファイル

会議フォルダごとに `meeting-flow.json` を作り、会議進行の正とする。

- `strategy.json`: 準備チャット履歴と準備ブリーフを保持する
- `meeting-flow.json`: 着地点、議題、質問、結果、AI提案を保持する
- `data.json`: 議事録の要約、論点、決定、ToDo、可視化用情報を保持する
- `transcript.txt`: 文字起こし本文を保持する

### 8.3 書き込み規律

- 会議IDを全要求で明示し、表示中会議のグローバル値に途中で依存しない
- 手動操作とAI更新は会議ごとのロック下で統合する
- 書き込みは一時ファイルから `os.replace` する
- すべての更新で `revision` を1増やす
- クライアントは旧 `revision` を付けて更新し、競合時は最新状態を取得し直す
- AIは全データの上書きを返さず、検証可能な差分操作だけを返す
- ユーザー作成・手動編集・却下済みの情報はAI更新より優先する

## 9. API仕様

### 9.1 取得

`GET /api/meeting-flow?sid=<meeting-id>`

応答:

```json
{
  "ok": true,
  "flow": {},
  "revision": 18
}
```

### 9.2 手動操作

`POST /api/meeting-flow/action`

```json
{
  "sid": "meeting-id",
  "revision": 18,
  "action": "agenda.reorder",
  "payload": {
    "agendaIds": ["agenda-2", "agenda-1", "agenda-3"]
  }
}
```

操作種類:

- `target.update`
- `agenda.create`
- `agenda.update`
- `agenda.delete`
- `agenda.reorder`
- `agenda.accept`
- `agenda.merge`
- `question.create`
- `question.update`
- `question.accept`
- `question.move`
- `question.reorder`
- `question.dismiss`
- `result.update`
- `suggestion.accept`
- `suggestion.defer`
- `suggestion.dismiss`

各操作は対象ID、許可された状態、文字数、会議IDをサーバー側で検証する。

`revision` が古い場合は `409` と最新 `flow` を返す。クライアントはユーザー入力を失わず、最新状態へ再適用する。

### 9.3 可視化の表示状態

現行のビューフォーカスAPIを拡張し、ボトムシートで実際に表示中のビューだけをアクティブとする。

## 10. AI差分操作

AIは次のような差分を返す。

```json
{
  "currentAgendaId": "agenda-uuid",
  "currentAgendaClientKey": "",
  "agendaStatusUpdates": [
    {
      "agendaId": "agenda-uuid",
      "status": "discussing",
      "basis": "料金プランの比較を開始した"
    }
  ],
  "agendaResolutionUpdates": [
    {
      "agendaId": "agenda-uuid",
      "status": "pending",
      "basis": "比較中で、まだ合意発言はない"
    }
  ],
  "questionUpdates": [],
  "resultUpdates": [],
  "agendaProposals": [],
  "questionProposals": []
}
```

サーバーは差分の適用前に次を検証する。

- 対象議題・質問が存在するか
- ユーザーロック済みの項目でないか
- 根拠文字範囲が現行文字起こしに存在するか
- 議題・質問の類似重複がないか
- 許可された状態遷移か
- 創作された人名・会社名・数値が含まれていないか

## 11. 既存データとの互換性

### 11.1 新規会議

作成時に空の `meeting-flow.json` を作成する。準備チャットは初回応答から同ファイルを更新する。

### 11.2 既存会議

`meeting-flow.json` がない場合、1回だけ移行する。

- `meta.json.goal` / `strategy.json.board.outcome` → `target`
- `strategy.json.board.questions` → 未分類の質問
- `data.json.agenda` → AI由来の仮議題
- `data.json.decisions` / `todos` / `open` → 関連付けできる範囲で議題結果の候補

移行時に判定できない関係を創作しない。関連付けできない質問・決定・ToDoは未分類として保持する。移行前のファイルは変更しない。

## 12. レスポンシブ・アクセシビリティ

- すべてのアコーディオン、ボトムシート、ドロワーはキーボードで操作できる
- 議題のドラッグ並べ替えに必ずボタン操作を併設する
- 議題トグルに `aria-expanded`、ボトムシートとドロワーに適切なダイアログ属性を付ける
- フォーカスを開いた要素へ移し、閉じた後は起点ボタンへ戻す
- ユーザーが入力・編集中の内容をポーリングで置き換えない
- 色だけで状態を表さない
- `prefers-reduced-motion` でボトムシートと強調表示の動画を抑制する

## 13. 非機能要件

- 手動の議題・質問操作はAI応答を待たず保存する
- 文字起こし到着から時系列表示までは、AIの完了を待たない
- AIが正常な場合、意味のある新発言は15秒以内を目標に議題・質問・結果へ反映する
- AI失敗中も録音、文字起こし、手動編集を停止しない
- 更新中にメイン画面を空状態にしない
- AI更新でカードの順番、スクロール位置、開閉状態を勝手に変えない
- AIが提案した人名・会社名・数値は、根拠がない限り確定事実として表示しない

## 14. 受け入れ基準

### 14.1 準備

- [ ] 壁打ちで明示した着地点が会議設計に反映される
- [ ] 「議題にしたい」と言った内容が正式議題になる
- [ ] 「聞きたい」と言った内容が対応議題の質問になる
- [ ] AI推測の議題・質問は仮作成と分かる
- [ ] 準備の議題・質問がコピー処理なしで本番へ引き継がれる

### 14.2 メイン画面

- [ ] 着地点、議題、質問、回答、結果、未解決が1画面で確認できる
- [ ] 議題をドラッグとボタンの両方で並べ替えられる
- [ ] 議題を開閉しても自動更新で開閉状態が戻らない
- [ ] ユーザーが並べた議題順をAIが勝手に変えない
- [ ] 各議題に次の質問、回答済み質問、現在の結果が表示される
- [ ] 汎用的な「結論を確定」ボタンがない
- [ ] ユーザーが修正した結果をAIが上書きしない

### 14.3 AI進行支援

- [ ] 議題未設定でも会話から仮議題が作られる
- [ ] 挨拶・相槌だけで議題が作られない
- [ ] 現在の議題が変わると、対応する議題が強調される
- [ ] 根拠発言がない質問を回答済みにしない
- [ ] 明示的な合意がない内容を決定事項にしない
- [ ] AI提案がユーザーの承認なしで正式議題・質問にならない
- [ ] 却下した提案が同じ根拠で繰り返されない

### 14.4 ボトムシートと右ドロワー

- [ ] 着地点の5ボタンから対応ビューのボトムシートが開く
- [ ] キャッシュを即時表示し、更新中も旧表示を残す
- [ ] シート内で5ビューを切り替えられる
- [ ] ボトムシートを閉じてもメインのスクロール位置が変わらない
- [ ] 「詳しく見る」で選択した議題の右ドロワーが開く
- [ ] 狭幅時にボトムシートと右ドロワーが重ならない
- [ ] ライブフェーズに固定フッターがない

### 14.5 保全・競合

- [ ] 会議切替中のAI応答が別会議の進行データを書き換えない
- [ ] AI更新とユーザーの並べ替えが同時でもどちらも失われない
- [ ] 文字起こしの世代が変わった場合、古い根拠範囲を現行本文の根拠として使わない
- [ ] 手動編集中の入力がポーリング再描画で消えない
- [ ] 既存会議の移行で元ファイルが変更・削除されない

## 15. 実装順序

### マイルストーン1: 進行データ基盤

- `meeting-flow.json` の読み書き、スキーマ正規化、原子保存
- 操作API、revision競合制御
- 既存データの保守的移行
- 会議ID固定と並行更新テスト

### マイルストーン2: 手動で使える新メイン画面

- 着地点セクション
- 議題アコーディオン
- 議題の並べ替え
- 議題に紐づく質問・回答・結果
- 右詳細ドロワー
- 固定フッターの廃止

### マイルストーン3: 準備壁打ちの反映

- 準備AI出力に議題と議題紐づき質問を追加
- 明示入力とAI推測の出所分離
- 会議設計の差分表示
- 準備・ライブ間の同一データ参照

### マイルストーン4: 会議中のAI進行支援

- 進行解析レーン
- 議題未設定時の仮議題生成
- 現在議題、回答、結果、未解決の根拠付き更新
- 追加質問・議題順・打開案の提案
- 失敗時の再試行と処理範囲カーソル

### マイルストーン5: 会話可視化ボトムシート

- 5ビューのボタン移設
- ボトムシートの開閉、リサイズ、全画面
- キャッシュ即時表示とアクティブビュー差分更新
- 時系列の上下ジャンプ維持

### マイルストーン6: 調査・打開案・成果物統合

- AI調査と議題・質問の紐づけ
- 行き詰まり検知
- 清書、PDF、マインドマップ、スライドへの議題結果反映
- 実会議での通し検証

## 16. 非対象

本仕様では次を行わない。

- 複数人による同時編集
- AIが音声で会議に割り込む自動司会
- 背景フォルダ外への無制限調査
- UIフレームワークの全面導入
- 既存の録音保全、文字起こし、清書の信頼性要件の緩和

## 17. 既存UI仕様との関係

`ui-redesign-spec.md` は2026-07-16時点の実装済み仕様として保持する。本文書はライブフェーズの情報設計と配置を上書きする次期仕様である。

- 準備ボードの質問配列 → 議題に紐づく構造化質問
